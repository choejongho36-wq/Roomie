# Roomie 프로젝트 — Codex 작업 규칙

이 파일은 Codex가 이 프로젝트에서 작업할 때 반드시 따라야 하는 규칙입니다.
세션 시작 시 자동으로 읽히므로, 별도로 안내하지 않아도 아래 규칙이 적용됩니다.

---

## 1. 작업 시작 전 파일 목록 승인 (가장 중요)

코드를 수정·생성·삭제하기 전에 반드시 다음 절차를 따른다.

1. 요청받은 작업을 분석하여 **변경 예정 파일 목록**을 작성한다.
   - 신규 생성
   - 수정
   - 삭제
   - 각 파일마다 변경 이유를 한 줄로 설명한다.
2. **사용자의 명시적인 승인 전까지는 실제 파일을 수정하지 않는다.**
3. 사용자가 "진행", "승인", "OK" 등의 의사를 밝히면 작업을 시작한다.
4. 사용자가 목록 변경을 요청하면 수정된 목록을 다시 제시하고 승인받는다.

### 출력 형식

```text
다음 파일들을 변경할 예정입니다.

[신규 생성]
- src/.../ProfileController.java
  - 프로필 조회 API 추가

[수정]
- src/.../ProfileService.java
  - 프로필 수정 로직 추가

[삭제]
- 없음

이대로 진행해도 될까요?
```

### 예외

다음 경우에는 승인 없이 진행 가능하다.

- 코드 조회 및 설명만 하는 경우
- 사용자가 특정 파일 하나만 지정하여 수정 요청한 경우
- 5줄 미만의 단순 수정(주석, 오탈자 등)

---

## 2. Git 규칙

- main 브랜치에 직접 커밋하거나 푸시하지 않는다.
- 현재 체크아웃된 브랜치에서만 작업한다.
- 커밋 메시지는 팀 컨벤션(feat:, fix:, docs:, refactor:, test:, chore:)을 따른다.
- 사용자의 명시적 요청 없이 `git push --force`를 사용하지 않는다.
- `git merge`, `git rebase` 등 히스토리를 변경하는 작업은 반드시 사전 승인을 받는다.

---

## 3. 코드 수정 원칙

- 기존 코드 스타일과 네이밍을 유지한다.
- 요청되지 않은 리팩터링은 수행하지 않는다.
- 설정 파일(application.yml, package.json, build.gradle 등)은 변경 이유를 명확히 설명한다.
- DB 관련 변경 시 현재 스키마와 일치하는지 확인한다.

---

## 4. 불확실한 경우

- 요구사항이 모호하면 구현하지 말고 선택지를 먼저 제시한다.
- 기존 기능과 충돌 가능성이 있으면 먼저 사용자에게 알린다.

---

## 5. 커밋 및 PR

- 변경 내용을 기준으로 커밋 메시지를 작성한다.
- PR 설명에는 변경 파일과 변경 이유를 함께 작성한다.

---

## 6. 명령 실행

- 사용자의 요청 없이 프로젝트를 빌드하거나 테스트하지 않는다.
- 데이터베이스 초기화, 마이그레이션, 파일 삭제 등 파괴적인 작업은 반드시 사전 승인을 받는다.
- 새로운 라이브러리 설치(npm install, gradle, pip 등)는 먼저 사용자에게 알린다.
- 여러 파일을 수정한 경우 작업 완료 후 변경 파일 목록을 요약한다.
