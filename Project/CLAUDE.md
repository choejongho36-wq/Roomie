# Roomie 프로젝트 — Claude Code 작업 규칙

이 파일은 Claude Code가 이 프로젝트에서 작업할 때 반드시 따라야 하는 규칙입니다.
세션 시작 시 자동으로 읽히므로, 별도로 안내하지 않아도 아래 규칙이 적용됩니다.

---

## 1. 작업 시작 전 파일 목록 승인 (가장 중요)

**코드를 수정/생성/삭제하기 전에, 반드시 다음 순서를 지킨다.**

1. 요청받은 작업을 분석해서 **건드릴 파일 목록**을 정리한다.
   - 새로 만들 파일 / 수정할 파일 / 삭제할 파일을 구분해서 표시
   - 각 파일에 대해 "왜 건드리는지" 한 줄 설명 포함
2. 목록만 출력하고, **사용자의 명시적 승인 전까지 실제 파일 수정을 시작하지 않는다.**
3. 사용자가 "진행해줘", "ok", "승인" 등으로 확인하면 그때 작업을 시작한다.
4. 사용자가 목록 중 일부를 빼거나 수정하라고 하면, 그 피드백을 반영한 새 목록을 다시 제시하고 재승인을 받는다.

**출력 형식 예시**

```
다음 파일들을 건드릴 예정입니다:

[신규 생성]
- src/main/java/.../ProfileController.java — 프로필 조회 API 컨트롤러 추가

[수정]
- src/main/java/.../ProfileService.java — 생활패턴 수정 로직 추가
- src/main/resources/application.yml — 프로필 이미지 업로드 경로 설정 추가

[삭제]
- (없음)

이대로 진행해도 될까요?
```

**예외**: 다음의 경우는 목록 승인 없이 바로 진행해도 된다.
- 단일 파일의 단순 조회/설명(코드를 읽고 답만 하는 경우)
- 사용자가 이미 특정 파일 하나를 정확히 지목하며 수정을 요청한 경우 (예: "ProfileService.java의 이 함수만 고쳐줘")
- 오탈자, 주석 수정처럼 5줄 미만의 사소한 변경

---

## 2. Git 안전 규칙

- **main 브랜치에 직접 커밋/푸시하지 않는다.** 항상 현재 체크아웃된 브랜치(feature 브랜치)에서만 작업한다.
- 커밋을 대신 만들 때는 [팀 커밋 컨벤션](./README.md)의 타입 접두사(`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:` 등)를 따른다.
- **`git push --force`는 사용자가 명시적으로 요청하지 않는 한 절대 사용하지 않는다.**
- 커밋 범위는 작업 단위로 잘게 나눈다. 여러 기능을 한 커밋에 몰아넣지 않는다.
- `git merge`, `git rebase`처럼 히스토리에 영향을 주는 명령은 실행 전 사용자에게 먼저 확인한다.

---

## 3. 파일 수정 시 원칙

- 기존 코드 스타일(들여쓰기, 네이밍 컨벤션)을 최대한 따른다. 임의로 전체 포맷을 바꾸지 않는다.
- 요청받지 않은 리팩토링(변수명 일괄 변경, 불필요한 구조 개편 등)은 하지 않는다. 필요하다고 판단되면 먼저 제안하고 승인을 받는다.
- 설정 파일(`application.yml`, `.env`, `build.gradle`, `package.json` 등)을 수정할 때는 특히 신중하게, 변경 사유를 목록에 명확히 적는다.
- 데이터베이스 스키마와 관련된 코드(엔티티 클래스 등)를 수정할 때는, 현재 DB 스키마(`roomie_schema.sql` 또는 최신 버전)와 어긋나지 않는지 확인한다.

---

## 4. 불확실한 경우

- 요구사항이 모호하거나 여러 구현 방법이 가능한 경우, 임의로 하나를 선택해 진행하지 않고 먼저 옵션을 제시하고 선택을 받는다.
- 이미 존재하는 기능/파일과 겹치거나 충돌할 가능성이 있으면, 진행 전에 이를 먼저 알린다.

---

## 5. 커밋/PR 관련 작업

- 커밋 메시지, PR 제목/설명을 작성할 때는 실제로 무엇이 바뀌었는지 기준으로 작성한다 (요청 문구를 그대로 복사하지 않는다).
- PR을 생성하는 경우, 어떤 파일이 왜 바뀌었는지 요약을 PR 설명에 포함한다.
